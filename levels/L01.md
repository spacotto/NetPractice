## How to Solve Level 1

### Goal 1: Host "my PC" needs to communicate with host "my little brother's computer"

To achieve Goal 1, we must connect `Host A` (my PC) to `Host B` (my little brother's computer). Because `Host B`'s interface (`B1`) is locked, we must modify `Host A`'s interface (`A1`) to match `Host B`'s network parameters. 

The **subnet mask** `255.255.255.0` dictates that the first three octets (`104.98.23`) represent the **Network ID** and must be identical on both machines. Only the last octet identifies the specific host. Therefore, `Host A`'s IP must start with `104.98.23.`, and end with any valid host number from `1` to `254` (excluding `12`, which `Host B` is already using).

* [x] **STEP 1:** Change the second octet of `Host A` from `93` to `98` to match the Network ID.
* [x] **STEP 2:** Correct the final octet (which may show an invalid placeholder like `339`) to any valid host number between `1` and `254` (except `12`). For example, change it to `10`.

**Example Solution Configuration:**
| Device | Interface | IP Address | Subnet Mask |
| :--- | :--- | :--- | :--- |
| **Host A (PC)** | `A1` | `104.98.23.10` | `255.255.255.0` |
| **Host B (Brother)** | `B1` | `104.98.23.12` | `255.255.255.0` |

### Goal 2: Host "my Mac" needs to communicate with host "my little sister's computer"

To achieve Goal 2, we must connect `Host C` (my Mac) to `Host D` (my little sister's computer). Because `Host C`'s interface (`C1`) is locked, we must modify `Host D`'s interface (`D1`) to match `Host C`'s network parameters. 

The **subnet mask** `255.255.0.0` dictates that only the first two octets (`211.191`) represent the **Network ID** and must be identical on both machines. The remaining two octets can vary. Therefore, `Host D`'s IP must start with `211.191.`. The last two numbers can be any valid combination, provided they do not exactly duplicate `Host C`'s full IP address.

* [x] **STEP 1:** Change the second octet of `Host D` from `190` to `191` to match the Network ID.
* [x] **STEP 2:** Correct the third octet (fixing any invalid placeholder like `282`) and the fourth octet to a valid value between `0` and `255`. For example, set the third octet to `100` and the fourth to `20`.

**Example Solution Configuration:**
| Device | Interface | IP Address | Subnet Mask |
| :--- | :--- | :--- | :--- |
| **Host C (Mac)** | `C1` | `211.191.161.1` | `255.255.0.0` |
| **Host D (Sister)** | `D1` | `211.191.100.20` | `255.255.0.0` |

## Theoretical Concepts

To successfully solve this level, you need to understand three core pillars of IPv4 addressing:

### 1. The Anatomy of an IPv4 Address
An IPv4 address consists of 32 bits divided into 4 sections called **octets** (8 bits each), separated by periods (e.g., `192.168.1.1`). 
* Because each octet is 8 bits, the values can only range from **0 to 255** in decimal notation. Any number outside this range is invalid.

### 2. The Role of the Subnet Mask
A subnet mask tells the computer where the "Network prefix" ends and the "Host identifier" begins. It acts as a blueprint for local communication:
* **`255.255.255.0` (or /24):** The first 3 octets are locked for the Network ID. Devices must share these 3 numbers exactly to see each other on a local switch without a router.
* **`255.255.0.0` (or /16):** Only the first 2 octets are locked for the Network ID. This allows for a much larger pool of local devices since the final two octets can be customized.

### 3. Basic Local Networking
When devices are connected to the same unrouted segment (like a simple hub or switch), they can only forward packets directly if they belong to the **exact same subnet**. If even one digit of the Network ID differs, the device will assume the destination is on an outside network and drop the packet unless a gateway/router is configured.
