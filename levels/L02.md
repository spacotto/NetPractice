## How to Solve Level 2

### Goal 1: Host "Computer B" needs to communicate with host "Computer A"

To achieve Goal 1, we must connect `Host A` (Computer A) to `Host B` (Computer B). `Computer A` has an IP of `192.168.141.1` with a mask of `255.255.255.224` (a `/27` network, meaning it can only talk to IPs from `.1` to `.30`). While `Computer B` has an IP of `192.168.141.222`, which is way outside A's network. Furthermore, its mask is set to `255.255.255.32`, which is a mathematically **illegal subnet mask**.

To make them communicate without changing their default IP addresses, we need to widen the subnet mask so that both `.1` and `.222` fall into the exact same network segment. A standard Class C mask (`255.255.255.0`) perfectly spans from `.1` to `.254`.

* [x] **STEP 1:** Change `Computer A`'s Subnet Mask to `255.255.255.0`.
* [x] **STEP 2:** Change `Computer B`'s Subnet Mask from the broken `255.255.255.32` to `255.255.255.0`.

*(Note: If the simulation forces you to keep Computer A's mask locked at `255.255.255.224`, you must instead change Computer B's IP to a valid host in A's range, such as `192.168.141.2`, and match its mask to `255.255.255.224`.)*

**Example Solution Configuration (Widened Mask Method):**
| Device | Interface | IP Address | Subnet Mask |
| :--- | :--- | :--- | :--- |
| **Computer A** | `A1` | `192.168.141.1` | `255.255.255.0` |
| **Computer B** | `B1` | `192.168.141.222` | `255.255.255.0` |

### Goal 2: Host "Computer D" needs to communicate with host "Computer C"

To achieve Goal 1, we must connect `Host C` (Computer C) to `Host D` (Computer D). `Computer C` is configured with IP `127.0.0.1` and Subnet Mask `255.255.255.252` (a `/30` mask). This creates an incredibly tight subnet block size of 4 addresses (`127.0.0.0` to `127.0.0.3`), where only `.1` and `.222` are usable host slots.
* `Computer D` is configured with IP `127.0.0.4` and mask `/30`. In a `/30` structure, `.4` represents the **Network ID** of the next block (`127.0.0.4` to `127.0.0.7`). A host can never use a Network ID as its interface IP, and it cannot talk across blocks directly.

We must bring `Computer D` backward into `Computer C`'s active network block. Since `.1` is already claimed by Computer C, the only remaining valid IP slot in this subnet is `.2`.

* [x] **STEP 1:** Change `Computer D`'s IP address from `127.0.0.4` to `127.0.0.2`.
* [x] **STEP 2:** Keep or verify its mask as `/30` (which perfectly mirrors `255.255.255.252`).

**Example Solution Configuration:**
| Device | Interface | IP Address | Subnet Mask |
| :--- | :--- | :--- | :--- |
| **Computer C** | `C1` | `127.0.0.1` | `255.255.255.252` |
| **Computer D** | `D1` | `127.0.0.2` | `/30` |

## Theoretical Concepts

### What Makes a Subnet Mask "Illegal"?

Subnet masks are evaluated in binary from left to right. They *must* consist entirely of continuous ones (`1`), followed entirely by continuous zeros (`0`). 
* `255.255.255.224` in binary is `11111111.11111111.11111111.11100000` (Valid! Continuous ones).
* `255.255.255.32` in binary is `11111111.11111111.11111111.00100000` (Invalid! The isolated `1` breaks the consecutive masking rule).

### Dotted Decimal vs. CIDR Slash Notation

NetPractice accepts both formats interchangeably, but you must know how to translate them in your head during evaluations:
* `/30` means the first 30 bits are locked for network routing, leaving 2 bits for hosts ($32 - 30 = 2$ bits).
* Converting the binary mask `11111111.11111111.11111111.11111100` to decimal yields `255.255.255.252`. They are completely identical in function.

### Understanding Block Sizes (The /30 Subnet)

When dealing with a `/30` (or `255.255.255.252`), your network space fragments into micro-blocks of 4 total addresses:

```text
[Block 1]
127.0.0.0 -> Network ID (Unusable)
127.0.0.1 -> Host A
127.0.0.2 -> Host B
127.0.0.3 -> Broadcast Address (Unusable)

[Block 2]
127.0.0.4 -> Next Network ID (Where Computer D was mistakenly sitting!)
