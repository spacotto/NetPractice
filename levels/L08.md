## How to Solve Level 8

Generally, we start at the lower layers of the network, then advance towards the Internet layer (Down-Top Planning). In this case, due to the peculiar topology and restrictions, we can follow the opposite strategy (Top-Down Planning).

### Fix Internet Routing Table

In this topology, the entire internal network must fit within a very specific IP block dictated by the Internet and Router 2. The `Internet` routing table has a locked destination route of `[b].[c].[d].0/26`. This means the Internet will only route traffic destined for addresses between `.0` and `.63` back to your internal network. Every internal device (Hosts C and D, and the internal router links) MUST share this `[b].[c].[d]` prefix and fit inside that 64-address limit. Fix the `Internet` Gateway to point back to Router 1's public interface (`R12`) by changing it to `163.[a].250.12`.

### Determine the R1-R2 Transit Link

Router 2 must have a default route to send outgoing traffic to Router 1, but its gateway is locked.
* Examine `Router R2`'s routing table (`R2r1`). The Gateway is permanently locked to `[b].[c].[d].62`. 
* Change the destination on `R2r1` to `0.0.0.0/0`.
* Because `R2` expects `R1` to be at `.62`, assign the IP `[b].[c].[d].62` to `Interface R13`.
* To isolate this transit link without wasting the limited IP space, apply a `/30` mask (`255.255.255.252`). A `/30` block ending at `.63` (the broadcast address) spans `.60` to `.63`. 
* Assign the other usable IP in this block (`[b].[c].[d].61`) to `Interface R21`, and apply the `/30` mask.

### Allocate Host Networks (VLSM)

With `.60` through `.63` consumed by the transit link, you have addresses `.0` through `.59` remaining to split between Host C and Host D.
* **Network D:** `Interface D1` has a locked mask of `255.255.255.240` (`/28`), requiring a block of 16 addresses. Allocate the `.16` block (spanning `.16` to `.31`). Assign `R23` the IP `.17` and `D1` the IP `.18`. Apply the `/28` mask to `R23`.
* **Network C:** Allocate the `.0` block (spanning `.0` to `.15`) to Host C. Assign `R22` the IP `.1` and `C1` the IP `.2`. Apply a `/28` mask (`255.255.255.240`) to both.

### Configure Routing Tables

Establish the remaining routes to ensure traffic flows seamlessly end-to-end.
* **Hosts C & D:** Point their default routes (`0.0.0.0/0`) to their respective `R2` gateways (`.1` and `.17`).
* **Router R1:** `R1` needs a route to send incoming Internet traffic back to `R2` for distribution to the hosts. Add a summary route of `[b].[c].[d].0/26` with the Gateway pointing to `R21` (`[b].[c].[d].61`).

## Theoretical Concepts

### Route Summarisation

Instead of maintaining a separate routing table entry for every single small subnet (like the `/28` blocks for C and D), Router 1 uses a single **summary route** (`[b].[c].[d].0/26`). This single entry covers all 64 addresses. When R1 receives traffic for Host C or Host D, the summary route catches it and forwards it to R2, drastically reducing routing table size and CPU overhead.

### Longest Prefix Match

You might notice a potential conflict on Router 1: its connected interface (`R13`) sits in the `.60/30` subnet, but its static routing table says the entire `.0/26` subnet lives behind R2. How does it know not to send `.62` traffic back to R2? 

Routers always use the **Longest Prefix Match**. Because `/30` is a more specific (longer) prefix than `/26`, the router prioritises the `/30` rule for the `.60-.63` IPs, and relies on the `/26` rule for everything else.
